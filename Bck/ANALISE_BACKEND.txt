1. VISÃO GERAL DO PROJETO
================================================================================

O FOCAplus é uma aplicação backend desenvolvida em Spring Boot que implementa
um sistema de gestão acadêmica e de estudos. O projeto utiliza arquitetura
modular seguindo princípios de Clean Architecture, com separação clara entre
camadas de infraestrutura, domínio e apresentação.

Tecnologias Principais:
- Spring Boot 3.2.8
- Spring Data JPA (Hibernate)
- Spring Security com JWT
- PostgreSQL (Banco de Dados)
- Lombok (Redução de boilerplate)
- SpringDoc OpenAPI (Swagger/OpenAPI)
- Spring Cloud OpenFeign (Comunicação entre serviços)
- Maven (Gerenciamento de dependências)

3. CONFIGURAÇÕES
================================================================================

3.1. APPLICATION.YML
- Porta do servidor: 8080 (configurável via WEBAPI_PORT)
- Banco de dados: PostgreSQL
  - URL padrão: jdbc:postgresql://localhost:5432/db_foca
  - Usuário e senha configuráveis via variáveis de ambiente
- JPA/Hibernate:
  - DDL Auto: update (cria/atualiza tabelas automaticamente)
  - Show SQL: true (para debug)
  - Dialeto: PostgreSQLDialect
- JWT:
  - Secret: configurável via JWT_SECRET
  - Expiração: 3600000ms (1 hora)
  - Refresh Token: 86400000ms (24 horas)
- Swagger/OpenAPI:
  - Path: /foca.html
  - API Docs: /foca
  - Scan de pacotes: com.focados.foca.modules

3.2. SECURITY CONFIG
- Autenticação: JWT (JSON Web Token)
- Política de sessão: STATELESS
- CSRF: Desabilitado (API REST)
- Endpoints públicos:
  - /api/v1/auth/** (login, registro, refresh)
  - /api/v1/users (criação de usuário)
  - /swagger-ui/** (documentação)
  - /v3/api-docs/**
- Endpoints protegidos:
  - /api/v1/** (requer autenticação JWT)

3.3. JWT AUTH FILTER
- Filtro de autenticação JWT
- Valida tokens Bearer no header Authorization
- Extrai userId do token e define no SecurityContext
- Continua a requisição mesmo se token inválido (sem autenticação)

5. PADRÕES ARQUITETURAIS
================================================================================

5.1. Clean Architecture / Hexagonal Architecture
O projeto segue princípios de Clean Architecture com separação em camadas:

- Domain Layer (Camada de Domínio):
  * Services: Contém a lógica de negócio pura
  * DTOs: Objetos de transferência de dados (Request/Response)
  * Mappers: Conversão entre entidades e DTOs

- Infrastructure Layer (Camada de Infraestrutura):
  * Database: Entidades JPA e Repositórios
  * HTTP: Controllers REST

- Application Layer (Camada de Aplicação):
  * FocaApplication: Classe principal Spring Boot
  * Config: Configurações globais (Security, Swagger, etc.)

5.2. Padrão Repository
Todos os módulos utilizam Spring Data JPA Repositories para acesso a dados,
seguindo o padrão Repository que abstrai a camada de persistência.

5.3. Padrão DTO (Data Transfer Object)
Separação clara entre entidades de domínio (Models) e objetos de transferência
(DTOs), garantindo que a estrutura interna não seja exposta diretamente na API.

5.4. Padrão Service
Cada módulo possui services que encapsulam a lógica de negócio, mantendo os
controllers leves e focados apenas em receber requisições e retornar respostas.

6. SEGURANÇA
================================================================================

6.1. Autenticação JWT
- Tokens JWT para autenticação stateless
- Access Token: 1 hora de validade
- Refresh Token: 24 horas de validade
- Secret configurável via variável de ambiente

6.2. Spring Security
- Filtro JWT customizado (JwtAuthFilter)
- Proteção de endpoints via SecurityFilterChain
- Senhas criptografadas com BCrypt

6.3. Endpoints Públicos vs Protegidos
- Públicos: Autenticação, registro, documentação
- Protegidos: Todos os demais endpoints requerem JWT válido

7. BANCO DE DADOS
================================================================================

7.1. PostgreSQL
- Banco de dados relacional PostgreSQL
- Configuração via variáveis de ambiente ou application.yml
- DDL Auto: update (cria/atualiza schema automaticamente)

7.2. Entidades Principais Identificadas:
- users (UserModel, RefreshTokenModel)
- courses (CourseModel, UserCourseModel)
- periods (PeriodTemplateModel, PeriodInstanceModel)
- materias/disciplines (DisciplineTemplateModel, DisciplineInstanceModel,
                        DisciplineTeacherModel, DisciplineScheduleModel)
- tasks (TaskModel, TaskCollaboratorModel)
- assessments (AssessmentModel, AssessmentCollaboratorModel)
- studySessions (StudySessionModel)
- selfEvaluation (SelfEvalQuestionModel, SelfEvalAnswerModel)
- score (ScoreRecordModel)
- friendships (FriendshipModel)
- rewards (RewardModel)

7.3. Relacionamentos
O sistema possui relacionamentos complexos entre entidades:
- Usuários podem ter múltiplos cursos (UserCourse)
- Cursos têm períodos (templates e instâncias)
- Períodos têm disciplinas (templates e instâncias)
- Disciplinas têm tarefas, avaliações e sessões de estudo
- Tarefas e avaliações podem ter colaboradores
- Usuários podem ter amizades entre si
